Option 4

Assess an existing artifact

You have already written a SOAP or architecture document. Jupiter reads it, applies structured gate checks across multiple architecture dimensions, and produces a findings report — specific gaps and recommendations, not generic feedback.

End-to-end flow
  1. Open a terminal in the Jupiter repository and launch Claude Code

    Navigate to where you cloned Jupiter, then run: claude

  2. Run /jupiter:start — type 4 to assess an existing artifact

    Jupiter presents a welcome menu on first run. Select option 4: "Assess an existing artifact".

  3. Choose how to provide your document

    Type A for a local file path, or B to paste a Confluence page URL. Jupiter fetches Confluence content automatically — no manual export needed.

  4. Provide the file path or Confluence URL

    Enter the full path to your SOAP or architecture document. Jupiter copies it into the assessment workspace so your original is untouched.

  5. Describe the engagement in 1–2 sentences

    Example: "Review the payments SOAP against GDPR requirements and our internal API gateway standards." This scopes the evaluation for Jupiter — include the audience and known constraints.

  6. Confirm the artifact type when Jupiter asks

    Jupiter reads the document and infers whether it is an Architecture/Design doc, Engineering Process doc, or Requirements/Analysis doc. Confirm or correct its inference — this determines which gate checks apply.

  7. Wait — Jupiter runs gate checks and writes the findings report

    The loop agent evaluates the artifact across all gate check dimensions and writes the report to workspace/artifacts/assessment/.../findings.md.

  8. Review the findings, then close with /jupiter:review

    Read the findings report. When satisfied, run /jupiter:review to formally record the assessment outcome and close the initiative.

What you get
  • Structured findings report with pass/fail across each gate check dimension
  • Specific gaps by category — design quality, NFR coverage, standards compliance, requirement traceability
  • Actionable recommendations for each finding, from a Domain Solution Architect perspective
  • Recorded assessment outcome in workspace/log.jsonl for traceability

Tips for your assessment session

Load guardrails for sharper findings

If you have architecture standards, a landscape map, or prior ADRs, load them into workspace/context/ before running. Jupiter uses these to evaluate the artifact against your organisation's actual constraints, not generic ones.

Assessments are isolated

Each assessment creates its own initiative ({parent}-assessment-001) inside the workspace. Your main initiative is unaffected. You can assess multiple SOAPs in sequence without interference.

Iterate to deepen the findings

After reading the first findings report, run /jupiter:iterate again to dig deeper into specific gaps. Jupiter re-reads the artifact with the previous findings in context and can surface additional detail.

No requirements baseline needed

Assessment does not require a prior requirements artifact. If you have one, add it to workspace/artifacts/requirements/ — Jupiter will cross-reference it. If not, Jupiter evaluates the design on its own merits.

Optional: watch progress in the web dashboard

Jupiter's companion dashboard shows gate check progress, the live activity log, and the findings report as it is written. Run: cd web && npm install && npm run dev, then open http://localhost:3000. Read-only — all actions still go through Claude Code commands.
