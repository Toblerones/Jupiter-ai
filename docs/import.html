Option 5

Import existing requirements

You have an existing Business Requirements Document. Jupiter normalises it into structured requirements, then writes the full SOAP and all Architecture Decision Records — with your review and approval at each phase gate.

Phase 1 — Requirements
Requirements flow
  1. Open a terminal in the Jupiter repository and launch Claude Code

    Navigate to where you cloned Jupiter, then run: claude

  2. Run /jupiter:start — type 5 to import existing requirements

    Select option 5: "Import existing requirements".

  3. Provide your BRD as a file path or Confluence URL

    Type A for a local file, or B to paste a Confluence URL. Jupiter fetches Confluence pages automatically and saves the content locally for traceability.

  4. Answer 5 project identity questions

    Jupiter asks one at a time: project name, business owner, lead architect, compliance obligations (e.g. GDPR, PCI-DSS), and any existing architecture constraints. Takes about 3 minutes.

  5. Jupiter normalises your requirements — read the gate report

    The loop agent reads your BRD and produces a structured requirements artifact at workspace/artifacts/requirements/ with formal REQ keys (F, NFR, BR, C, S). A gate report shows which checks pass and which gaps remain.

  6. Run /jupiter:iterate if there are gaps — or approve via /jupiter:review

    If the gate report shows open gaps, run /jupiter:iterate to address them. When gap = 0 and you are satisfied with the requirements, run /jupiter:review and approve. Jupiter waits for your explicit sign-off before proceeding to the design phase.

Phase 2 — Design
Design flow
  1. Jupiter writes the SOAP and all ADRs

    The loop agent loads your approved requirements and produces the full Solution On A Page — every section with substantive content, not placeholders — plus all Architecture Decision Records with rationale and options considered. This begins automatically after requirements approval.

  2. Run the panel review — /jupiter:review --panel

    Five reviewer agents each assess the SOAP from their domain: Enterprise Architect, Business Architect, Data Architect, Solution Architect, and Engineering Lead. Their consolidated findings appear as a structured review report for you to consider before approving.

  3. Approve the design — /jupiter:review

    Read the panel report and make the call. Run /jupiter:review to record your approval and close the design phase. Your completed SOAP is at workspace/artifacts/design/ and ADRs at workspace/artifacts/design/adrs/.

What you get
  • Structured requirements artifact with formal REQ keys (Functional, NFR, Business Rule, Compliance, Security) and full traceability
  • Full Solution On A Page (SOAP) — every section written by Jupiter from your approved requirements
  • All Architecture Decision Records with rationale, options considered, and ratified status
  • Multi-domain reviewer report covering enterprise, business, data, solution, and engineering perspectives

Tips for your import session

Load architecture context as guardrails

When Jupiter asks during setup, load existing standards docs, landscape maps, and prior ADRs. Jupiter uses these when writing the SOAP — the more context loaded, the more organisation-specific the output.

Human gates are yours to own

Jupiter never auto-approves. When it says "ready for review", read the artifact. Run /jupiter:iterate to address gaps before approving. Run /jupiter:review when satisfied. This is true for both the requirements gate and the design gate.

The panel is advisory — you decide

The five reviewer agents (EA, BA, DA, SA, Engineering Lead) each produce a structured report. Read all five, weigh the findings, and then decide. Run /jupiter:review to record your decision. Approving with open reviewer concerns is allowed — just be deliberate.

Check traceability before handoff

After the design is approved, run /jupiter:gaps to verify every REQ key in requirements is addressed in the SOAP. Then run /jupiter:handoff to generate the architecture handoff package.

Optional: watch progress in the web dashboard

Jupiter's companion dashboard shows phase progress, gate check breakdown, live activity log, and rendered artifact content as Jupiter works. Run: cd web && npm install && npm run dev, then open http://localhost:3000. Read-only — all actions still go through Claude Code commands.
